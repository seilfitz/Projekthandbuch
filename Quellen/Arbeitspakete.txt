# 11 Arbeitspakete – Grobstruktur

## 11.1 Ziel

Die Arbeitspakete bilden die Grundlage für:

- Aufwandsschätzung
- Kostenaufstellung
- Projektplan
- Controlling
- Abrechnung einzelner Leistungen

Jedes Arbeitspaket erhält später:

```text
ID
Titel
Beschreibung
Typ
Priorität
Aufwand Entwicklung
Aufwand Test
Aufwand Dokumentation
Risiko
Abhängigkeiten
```

# Das fällt mir jetzt auf

Ich glaube, wir sollten die Arbeitspakete **nicht nur nummerieren**, sondern zusätzlich klassifizieren.

Beispielsweise:

| Kategorie | Beschreibung |
|------------|--------------|
| Architektur | Infrastruktur |
| Backend | REST / Services |
| Datenbank | Oracle |
| Portal | Blazor |
| Integration | SAP / Mail |
| Migration | WPF |
| Test | QS |
| Dokumentation | Projektdokumentation |

Warum?

Nehmen wir beispielsweise:

**AP-006.002 Abteilungsverwaltung**

Das besteht eigentlich aus:

- Oracle
- REST
- Blazor
- Test

Wenn wir das nicht trennen, lässt sich später der Aufwand nur schwer kontrollieren.

# Ich glaube, das wird später einer der größten Vorteile unseres Projekthandbuches

Denn daraus entstehen automatisch:

- Aufwand je Datenbank
- Aufwand REST
- Aufwand Blazor
- Aufwand Tests
- Aufwand Dokumentation

**ohne dass wir später alles erneut zerlegen müssen.**

## Ich würde deshalb die Arbeitspakete konsequent **dreistufig** aufbauen:

- **AP-006** → Fachpaket (z. B. Organisation)
- **AP-006.002** → Funktion (z. B. Abteilungsverwaltung)
- **AP-006.002.01** → Technische Umsetzung (DB, REST, Portal, Test)

Das halte ich insbesondere für dein Projekt für ideal, weil du als einzelner Entwickler bzw. kleines Team sehr genau nachvollziehen kannst, **wo** der Aufwand tatsächlich entsteht. Gleichzeitig erhält dein Auftraggeber auf Wunsch nur die obere Ebene (Fachpakete), während du intern bis auf die technische Umsetzungsebene herunterplanen kannst. Ich glaube, genau diese Struktur wird später die belastbarste Grundlage für Planung, Controlling und Nachkalkulation sein.

# 11.23 AP-019 WPF-Migration

Hier möchte ich bewusst eine andere Struktur wählen.

Die Migration erfolgt **nicht modulweise**, sondern **Service für Service**.

Das passt wesentlich besser zu deiner Strategie.

## Migrationsreihenfolge

| AP | Beschreibung |
|---|---|
| AP-019.001 | Authentication |
| AP-019.002 | Stammdaten |
| AP-019.003 | Referenzdaten |
| AP-019.004 | Dokumente |
| AP-019.005 | Rechnungen |
| AP-019.006 | Kalender |
| AP-019.007 | Freie Zeiten |
| AP-019.008 | Buchungen |
| AP-019.009 | Gebühren |
| AP-019.010 | Administration |

---

## Warum diese Reihenfolge?

Sie minimiert das Projektrisiko:

1. **Zuerst lesende Funktionen** (geringes Risiko).
2. **Danach komplexere Fachfunktionen**.
3. **Schreibende Funktionen** erst nach erfolgreicher Stabilisierung der REST-API.

Ich halte diese Reihenfolge für deutlich sinnvoller als eine Migration nach WPF-Modulen, weil sie exakt deiner Zielarchitektur entspricht und jederzeit einen stabilen Zwischenstand ermöglicht. Das wird später auch den Rollout beim Eigenbetrieb erheblich vereinfachen und Risiken im Produktivbetrieb reduzieren.
